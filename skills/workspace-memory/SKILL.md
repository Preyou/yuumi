# Workspace Memory

使用这个技能在工作目录根下维护 `.memory/`，把容易因上下文压缩丢失、但又会影响后续工作的关键信息落到磁盘。

## 何时使用

- 任务会持续很多轮，且中间结论很多。
- 已经发现重要约束、决策、排障线索或待办，后续很可能继续依赖。
- 需要给未来的自己保留 handoff，上下文一旦被压缩就会影响连续性。

## 目录约定

- `.memory/` 按需懒创建，不必预先铺满结构。
- 当前会话必须先解析自己的唯一标识；在 Codex Desktop 环境中默认使用 `CODEX_THREAD_ID`。
- 当前会话的专属记忆文件命名为 `.memory/<thread-id>.md`。
- 若当前会话文件尚不存在，可按需懒创建；后续持续写入同一个文件，而不是再创建共享可写文件。
- `.memory/` 中其他文件视为并行会话的记忆文件。

## 写入原则

- 只记录高价值信息：事实、结论、约束、下一步、未决问题。
- 优先写短摘要，不把大段日志或整份源码复制进去。
- 只写入当前会话自己的专属记忆文件。
- 若参考了其他记忆文件中的内容，应把需要复用的部分摘录或总结到自己的专属文件，而不是直接改写对方文件。
- 当前会话文件中的信息过时后及时改写或删除，避免陈旧记忆误导后续工作。
- 如果某条信息已经进入正式源码、文档或规则文件，可从当前会话文件中删掉或缩写成索引。

## 自主管理权限

- AI 可以不经询问地创建、修改、删除当前会话自己的专属记忆文件。
- AI 可以在任务推进过程中多次整理自己的专属文件，例如合并重复笔记、清理过期记录、重写摘要。
- 当前会话文件名保持稳定为 `.memory/<thread-id>.md`，不要重命名成其他路径。
- AI 可以读取其他 `.memory/*` 文件，但不得修改、重命名、删除它们。
- 除非用户明确要求，不必把 `.memory/` 的每次内部变动都当作正式交付内容汇报。

## 边界

- 不把 `.memory/` 当作用户配置、业务数据或正式交付物的永久存储。
- 不在子目录重复创建多个 `.memory/`；统一使用工作目录根下的 `.memory/`。
- 不创建跨会话共享可写的 memory 文件。
- 若用户明确要求某类敏感信息不要写入磁盘，则不要写入 `.memory/`；必要时只记录引用位置或简短提示。

## 读取策略

- 优先读取当前会话自己的专属记忆文件。
- 只读取与当前任务相关的其他 `.memory/*` 文件，不要每次全量加载整个目录。
- 其他会话文件仅作只读参考；如果需要采用其中信息，把结论同步到自己的专属文件。
- 开始复杂任务前，先检查是否已有可复用记忆。
- 任务结束时，按需要保留最小可复用信息，删除纯临时噪音。
